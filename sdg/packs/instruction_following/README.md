# Instruction Following Pack

`instruction_following` is a general pack for verifiable IFBench-style instruction-following rows.

The pack uses one expanded benchmark family:

- IFBench-style rows with deterministic constraints, multilingual prompt surfaces, and both single-turn and multiturn isolation variants

## What the pack does

Each row has:

- a structured hidden constraint list
- a visible instruction block rendered from those constraints
- a natural scenario generated by an LLM
- deterministic strict and loose verification against the hidden constraints

The verifier does not rely on a judge model.
The visible instruction block is intentionally varied across rows, so the pack does not always present constraints under the same fixed heading.

Prompt surfaces can come from either:

- the built-in synthetic topic/scenario pools
- an external prompt reservoir, where we take user prompts only, ignore source responses, then add fresh verifiable instructions on top

## Current scope

- languages: English and Danish
- interaction styles: single-turn and multiturn isolation
- response shapes:
  - plain text
  - numbered list
  - bullet list
  - JSON object
  - XML object
  - two separated responses
  - indented multi-line output

## LLM usage

The pack uses `models.scenario_writer` to generate the conversational scaffold around the deterministic constraint block.

When a prompt reservoir is configured:

- single-turn rows use the source user prompt directly and append the deterministic instruction block
- multiturn rows use the source user prompt as the first user turn, then use `models.scenario_writer` to answer that prompt normally before adding a constrained rewrite turn
- source responses are never reused

If you also want target responses attached during build, set:

- `generation.attach_targets: true`
- `models.answer_teacher`

Those responses are verified with the same deterministic checker layer. When targets are attached, `messages` includes the final assistant turn so the row is a full conversation.
For this pack, `answer_teacher` generation defaults to `temperature: 0.7` unless a config overrides it, so low-quality outputs can be filtered rather than suppressed upfront.

The `ifbench_laerebogen_da.yaml` config now does this by default, so Danish `laerebogen` runs produce full conversations and can be checked with strict and loose response verification.

## Start here

Run English IFBench-style single-turn rows:

```bash
uv run sdg build sdg/packs/instruction_following/configs/ifbench_single_turn.yaml
```

Run Danish IFBench-style single-turn rows:

```bash
uv run sdg build sdg/packs/instruction_following/configs/ifbench_single_turn_da.yaml
```

Run English IFBench-style multiturn rows:

```bash
uv run sdg build sdg/packs/instruction_following/configs/ifbench.yaml
```

Run Danish IFBench-style multiturn rows:

```bash
uv run sdg build sdg/packs/instruction_following/configs/ifbench_da.yaml
```

Run a mixed English/Danish multiturn set:

```bash
uv run sdg build sdg/packs/instruction_following/configs/ifbench_mixed.yaml
```

Run Danish multiturn rows seeded from `danish-foundation-models/laerebogen`:

```bash
uv run sdg build sdg/packs/instruction_following/configs/ifbench_laerebogen_da.yaml
```

That `laerebogen` config is now single-turn-heavy by default, while still keeping some multiturn rows in the mix.

## Prompt Reservoirs

Prompt reservoirs are configured under `generation.prompt_sources`, keyed by language.

You can also bias the prompt mix with `generation.interaction_style_weights`, for example to keep multiturn rows around while focusing mostly on single-turn.

The current loader supports:

- local JSONL files via `path`
- Hugging Face datasets via `dataset`

For conversation datasets like `laerebogen`, the loader expects a `messages` field and extracts user turns only. It deduplicates prompts, applies light filtering to avoid prompts that already hard-code rigid output formatting or existing requirement blocks, and balances source prompts by heuristic task type rather than sampling uniformly at random.

Prompt seeds are annotated with:

- `prompt_task_type`
- `prompt_length_bucket`

so you can inspect how much source diversity a run actually achieved.
